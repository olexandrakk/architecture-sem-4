# Аналіз лабораторної роботи №5 та Ретроспектива курсу

---

## 1. Аналіз лабораторної роботи №5: Модульний моноліт

### 1.1. Визначення меж контекстів та альтернативи
У межах цієї роботи монолітну систему Cinema API було фізично та логічно розділено на два основні обмежені контексти (Bounded Contexts):
* **Core (Основний):** Включає критичну бізнес-логіку - управління користувачами, фільмами, сеансами та процесом бронювання. Межі визначено за принципом функціональної цілісності: сутності, що потребують негайної узгодженості даних, об'єднані в один модуль.
* **Analytics (Аналітичний):** Допоміжний модуль для збору метрик та статистики продажів. Його відокремлення дозволяє використовувати власну таблицю в БД та іншу модель даних (агрегати та проєкції), яка не впливає на продуктивність основної системи.

**Альтернативи:** Розглядалася можливість виділення окремого контексту *Identity/Auth*. Проте для поточного масштабу проєкту це призвело б до надмірної складності (overengineering). Було прийнято рішення залишити авторизацію в `Core`, зосередившись на виділенні аналітики як найбільш незалежної частини.

### 1.2. Практична реалізація: Публічні контракти та ACL
Для забезпечення повної ізоляції модулів ми впровадили патерн **Public Contract**. Модулі спілкуються один з одним виключно через файли `index.js`, які віддають назовні лише необхідні роути або абстракції. Прямі імпорти з нутрощів іншого модуля (наприклад, звернення до чужих репозиторіїв) суворо заборонені.

Антикорупційний шар (Anti-Corruption Layer) реалізований у модулі Analytics за допомогою класу `AnalyticsSubscriber`. Він захищає модуль від змін у Core:
* **Трансляція моделей:** Subscriber слухає загальносистемну подію `BookingCreated`, "відрізає" конфіденційні дані користувача (email, ПІБ) і мапує корисне навантаження у внутрішню спрощену модель (назва фільму та ціна).
* **Ізоляція домену:** Якщо структура таблиці `bookings` у Core зміниться, оновленню підлягатиме лише цей транслятор у шарі ACL, а не вся бізнес-логіка аналітики.

### 1.3. Консистентність та підготовка до мікросервісів
У системі реалізовано комбінований підхід до консистентності:
* **Strong Consistency (Сувора узгодженість):** Працює у межах модуля Core (бронювання місця), щоб гарантувати відсутність подвійних продажів одного й того ж квитка.
* **Eventual Consistency (Узгодженість у кінцевому підсумку):** Працює між Core та Analytics через події. Статистика оновлюється асинхронно, що прийнятно для бізнесу, оскільки затримка у звітах на мілісекунди не є критичною.

Завдяки такій структурі, для виділення Analytics у повноцінний мікросервіс знадобиться лише заміна in-memory `EventEmitter` на надійний брокер повідомлень.

## 2. Спільна ретроспектива курсу (Лабораторні роботи №1–5)

### 2.1. Еволюція архітектури Cinema API
Протягом курсу наш проєкт еволюціонував від найпростішого додатку до складної Enterprise-системи:
1.  **Лаб 1-2 (Базовий CRUD):** Початкова реалізація мала високу зв'язність (Tight Coupling). Контролери напряму працювали з базою даних, бізнес-логіка була розмита.
2.  **Лаб 3 (CQRS):** Впровадження Command Query Responsibility Segregation. Розділення логіки на Commands (запис) та Queries (читання) прибрало "Spaghetti code" із сервісів і зробило кожну операцію атомарною та легкою для тестування.
3.  **Лаб 4 (Event-Driven Architecture):** Виділення побічних ефектів. Ми впровадили `EventBus`, що дозволило розвантажити основний потік виконання. **Головне досягнення:** ізольованість збоїв (Fault Tolerance). Під час тестування ми довели, що падіння поштового сервісу (`NotificationService`) більше не ламає процес створення бронювання для користувача.
4.  **Лаб 5 (Modular Monolith):** Фізичне розділення коду на незалежні бізнес-контексти з власними базами даних та контрактами.

### 2.2. Trade-offs (Компроміси) та отриманий досвід
* **Що було б зроблено інакше:** Знаючи кінцевий результат (Модульний моноліт), ми б від самого початку використовували структуру директорій **"Folder by Feature/Module"**, а не розкладали б файли за технічними шарами (всі контролери в одну папку, всі репозиторії - в іншу). Це позбавило б нас потреби у масовому та болючому рефакторингу шляхів імпортів на 5-му етапі.
* **Простота та Гнучкість:** Впровадження CQRS та Event Bus значно збільшило кількість файлів (Boilerplate code) і підвищило поріг входження в проєкт. Простий код пишеться швидше, але модульна архітектура значно полегшує підтримку команди розробників у довгостроковій перспективі.
* **Продуктивність та Складність:** Eventual Consistency забезпечує блискавичну відповідь API для користувача, але вимагає від нас додаткових зусиль для моніторингу цілісності даних (наприклад, обробки дублювання подій).

## 3. Загальний висновок

В ході виконання циклу лабораторних робіт ми пройшли повний шлях трансформації програмного забезпечення. Ми на практиці переконалися, що Чиста Архітектура (Clean Architecture) - це не просто набір папок, а спосіб мислення, який дозволяє захистити ядро бізнес-логіки від зовнішніх залежностей (фреймворків, баз даних, сторонніх API). 

Перехід до Модульного моноліту став ідеальним фіналом: ми отримали систему, яка поєднує в собі простоту розгортання моноліту (один сервер, один репозиторій) із масштабованістю мікросервісів (незалежні контексти, слабка зв'язність, обмін подіями). Cinema API повністю готовий до високих навантажень та подальшого розширення.
