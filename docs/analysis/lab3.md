# Аналіз впровадження патерну CQS (Лабораторна робота №3)

## Що змінилося в структурі проєкту порівняно з лабораторною роботою №2?
Основною зміною став рефакторинг шару застосунку (Application Layer). Замість універсальних Use Cases ми розділили операції на два типи:
1. **Команди (Commands)** — для операцій запису (створення бронювання, реєстрація).
2. **Запити (Queries)** — для операцій читання (отримання списку квитків, сеансів).

Кожна операція тепер має свій окремий об'єкт-команду/запит (DTO) та відповідний обробник (Handler). Контролери в шарі Presentation стали максимально «тонкими»: вони лише створюють об'єкти Command/Query з HTTP-запиту та передають їх обробникам.

## Переваги розділення на Command та Query
- **Різні вимоги до коду:** Ми змогли оптимізувати операції читання. Запити тепер можуть звертатися до бази даних напряму (через pool), минаючи складний доменний шар, що підвищує продуктивність.
- **Розділення відповідальності (SRP):** Кожен Handler відповідає лише за одну дію, що полегшує розуміння та підтримку коду.
- **Гнучкість тестування:** Команди тестуються через Unit-тести (перевірка бізнес-правил), а запити — через інтеграційні тести (перевірка SQL та мапінгу).

## Недоліки та ускладнення CQS
- **Зростання кількості класів:** Для кожної дії тепер потрібно створювати мінімум два класи (Command/Query + Handler).
- **Складність навігації:** Пошук логіки конкретної операції вимагає проходження через більше рівнів вкладеності папок.
- **Дублювання коду:** Іноді структури Command та відповідні валідатори можуть здаватися надлишковими для дуже простих операцій.

## Різниця між Command/Query Handler та Service
Класичний Service часто накопичує в собі десятки методів для читання та запису, перетворюючись на «божественний об'єкт» (God Object). Зміна одного методу в сервісі може випадково вплинути на інші або вимагати перевизначення багатьох залежностей.
CommandHandler/QueryHandler слідують принципу єдиної відповідальності. Якщо нам потрібно додати нову функцію (наприклад, скасування бронювання), ми просто створюємо нову пару Command/Handler, не торкаючись існуючого коду. Це значно покращує **розширюваність** (Open/Closed Principle).

## Структура даних запиту (Read Model) проти Доменної моделі
У нашій реалізації Запити повертають спеціальні **Read Models (DTO)**, а не доменні сутності.
**Чому це важливо?**
1. **Продуктивність:** Ми повертаємо лише ті поля, які потрібні клієнту (наприклад, назву фільму замість ID), уникаючи зайвих перетворень.
2. **Безпека:** Ми не ризикуємо «засвітити» внутрішні поля доменної моделі (наприклад, хеші паролів або технічні прапорці), які не повинні виходити за межі API.
3. **Незалежність:** Зміни в бізнес-логіці сутності не ламають інтерфейс читання, якщо структура DTO залишається стабільною.
