# Порівняльний аналіз архітектурних підходів (Лабораторна робота №4)

## 1. Що змінилося порівняно з лабораторною 3?

У попередній лабораторній роботі (де реалізовувався патерн CQRS) команди відповідали за повний цикл виконання бізнес-операції. Обробники команд (`CreateBookingCommandHandler` та `RegisterUserCommandHandler`) містили пряму залежність від інфраструктурних сервісів (`NotificationService` та `WelcomeEmailService`).

Потік виконання був лінійним і жорстко зв'язаним:

```text
Контролер -> Команда -> Збереження в БД -> Відправка листа -> Повернення результату клієнту
```

У поточній (4-й) лабораторній роботі було здійснено перехід до **Event-Driven Architecture (EDA)**.

Замість прямого виклику сервісів, обробники команд тепер лише:

- фіксують зміну стану системи;
- зберігають дані;
- публікують доменні події (`BookingCreatedEvent`, `UserRegisteredEvent`) через абстракцію `EventBus`.

Логіка відправки сповіщень була винесена в окремих підписників (**Subscribers**), які асинхронно реагують на ці події.

## 2. Які побічні операції ви виділили і чому?

Для асинхронної обробки було виділено:

- відправку вітального листа при реєстрації;
- відправку підтвердження бронювання на email.

Ці операції є класичними **side-effects** з таких причин:

### Не критичні для транзакції

Успішність створення квитка або акаунту в базі даних не повинна залежати від доступності SMTP-сервера.

Навіть якщо лист не відправився -бізнес-операція все одно успішно завершена.

### Повільний I/O процес

Взаємодія з мережею та сторонніми API займає час.

У лабораторній роботі це імітувалося затримкою у 2 секунди, що неприйнятно для блокування клієнта.

### Незалежність бізнес-домену

Доменна логіка завершується в момент:

- створення користувача;
- або закріплення місця за користувачем.

Інформування користувача -це вторинна сервісна функція (**eventual consistency**).

## 3. Порівняння синхронного та асинхронного підходів

| Критерій | Синхронний підхід (Лаб. 3) | Асинхронний підхід (Лаб. 4) |
|---|---|---|
| **Час відповіді API** | ~2100+ ms. Клієнт чекає завершення відправки листа перед отриманням `201 Created`. | ~10–50 ms. Відповідь повертається одразу після запису в БД. |
| **Поведінка при збоях** | Каскадний збій. Якщо поштовий сервіс недоступний - весь запит завершується помилкою `500`. | Ізольованість збоїв. Користувач все одно отримує акаунт або квиток. |
| **Зв'язаність (Coupling)** | Висока (`Tight Coupling`). Handler залежить від сервісів нотифікацій. | Низька (`Loose Coupling`). Handler лише публікує подію. |
| **Складність реалізації** | Нижча. Лінійний потік виконання, простіший debug. | Вища. Потрібен `EventBus`, асинхронний потік виконання. |
| **Тестування** | Потрібно мокати зовнішні сервіси. | Handler тестується лише на факт виклику `emit`. |

## 4. Який підхід варто обрати для production-системи і чому?

Для production-системи доцільніше використовувати **асинхронний підхід (Event-Driven Architecture)**, особливо для:

- мікросервісної архітектури;
- високонавантажених веб-додатків.

### Причини

Синхронне блокування потоку у Node.js призводить до:

- швидкого вичерпання ресурсів;
- блокування Event Loop;
- погіршення User Experience.

Асинхронність дозволяє:

- швидше відповідати клієнту;
- ізолювати помилки;
- масштабувати систему незалежно.

### Зауваження до поточного рішення

У лабораторній роботі використовується вбудований `EventEmitter`.

Це підходить для демонстрації концепції, але має ризики:

- втрата подій при падінні сервера;
- відсутність гарантій доставки;
- відсутність persistence.

У production-системах зазвичай використовують:

- RabbitMQ;
- Apache Kafka;
- AWS SQS.

Такі брокери гарантують:

- надійне збереження подій;
- повторну доставку;
- fault tolerance.

## 5. Що станеться, якщо подія буде доставлена двічі? Чи ідемпотентні обробники?

У distributed systems зазвичай використовується модель доставки:

```text
At-Least-Once Delivery
```

Це означає, що одна й та сама подія може бути доставлена повторно.

### Наслідки дублювання

У поточній реалізації:

- `BookingCreatedEvent` може бути оброблений двічі;
- `NotificationService` викличеться двічі;
- користувач отримає два однакових листи.

### Ідемпотентність

Поточні обробники **не є ідемпотентними**, оскільки:

- вони не перевіряють історію обробки;
- кожен виклик повторно виконує side-effect.

### Як вирішити проблему

Для забезпечення ідемпотентності можна реалізувати механізм дедуплікації.

#### Основні підходи

1. Генерувати унікальний `eventId` для кожної події.
2. Реалізувати `Inbox Pattern`.

### Приклад Inbox Pattern

Перед обробкою події Subscriber:

1. перевіряє таблицю `processed_events`;
2. шукає `eventId`;
3. якщо подія вже була оброблена - ігнорує її.

Таким чином side-effect буде виконано лише один раз.

## Висновок до ЛР №4

У ході виконання лабораторної роботи ми повністю переосмислили архітектуру нашого додатка, відмовившись від прямолінійного синхронного виконання на користь подійно-орієнтованого підходу.

Головним досягненням стало впровадження шини подій (`Event Bus`), яка дозволила розірвати жорсткі зв'язки між бізнес-логікою та інфраструктурними сервісами.

Це дало значний приріст у швидкості роботи API:

- було усунуто двосекундні затримки при реєстрації та бронюванні;
- сервер більше не очікує завершення відправки листа;
- клієнт миттєво отримує підтвердження успішного виконання операції.

Паралельно з цим було покращено безпеку системи:

- виправлено механізм роботи з JWT-токенами;
- реалізовано коректне хешування паролів;
- забезпечено правильну ідентифікацію користувачів.

У результаті ми отримали:

- масштабовану архітектуру;
- ізольованість збоїв;
- підвищену продуктивність;
- кращий користувацький досвід (UX);
- готовність системи до реальних навантажень.

Тепер збій у сервісі сповіщень більше не впливає на основний процес покупки квитків або реєстрації користувача, що відповідає сучасним стандартам розробки високонавантажених систем.
