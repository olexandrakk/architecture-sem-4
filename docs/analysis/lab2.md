Аналітичний звіт: Рефакторинг за принципами Чистої Архітектури
==============================================================

### Аналіз переходу від монолітної структури до Clean Architecture (Лабораторна робота №2).

1\. Порівняння структури проєкту (Лабораторна №1 та Лабораторна №2)
-------------------------------------------------------------------

У першій лабораторній роботі ми використовували підхід монолітного MVC з так званим "Fat Controller" (Товстий контролер). Весь код — від валідації вхідних даних до SQL-запитів та хешування паролів — знаходився безпосередньо в методах контролера Express. Проєкт був повністю рефакторений у 4-шарову архітектуру.

### Ключові відмінності:

*   **Розподіл відповідальності:** Замість одного файлу ми отримали чотири чітко визначені шари: Domain, Application, Infrastructure та Presentation. Кожен клас тепер має єдину відповідальність (Single Responsibility Principle).
    
*   **Напрямок залежностей:** У Лаб 1 контролер напряму залежав від конкретної бази даних. У Лаб 2 база даних стала лише деталлю реалізації, яка залежить від доменного інтерфейсу. SQL-запити повністю прибрані з контролерів.
    
*   **Локалізація логіки:** Бізнес-правила (формат email, перевірка унікальності) тепер винесені у Value Objects та Domain Factories, а контролери займаються виключно обробкою HTTP-запитів.
    

2\. Переваги розділення на шари
-------------------------------

1.  **Тестованість:** Головна перевага - ізоляція бізнес-правил. Тепер ми можемо тестувати логіку застосунку, взагалі не підключаючись до PostgreSQL чи Express. Ми просто замінюємо реальний репозиторій на заглушку, завдяки чому тести домену виконуються миттєво.
    
2.  **Гнучкість:** Ми відділили технічні рішення (яку бібліотеку хешування обрати, яку БД використовувати) від бізнес-вимог системи.
    
3.  **Повторне використання:** Логіка (наприклад, реєстрація) тепер не прив'язана до HTTP-запитів. Ми можемо викликати той самий UseCase з консольної команди або з черги повідомлень, не змінюючи жодного рядка коду.
    

3\. Недоліки та ускладнення
---------------------------

*   **Збільшення об'єму коду:** Для простої операції замість 20 рядків у контролері тепер потрібно створити окрему сутність, фабрику, інтерфейс репозиторію, його реалізацію та Use Case. Додатково довелося впроваджувати механізм мапінгу доменних помилок у HTTP-статуси.
    
*   **Складність навігації та відстеження коду:** Через високий рівень абстракції лінійний потік виконання програми втрачається. Замість читання одного методу згори донизу, доводиться стрибати по ланцюжку делегувань. Це підвищує навантаження та ускладнює дебагінг.
    
*   **Надмірність:** Для маленьких проєктів такий підхід може здатися занадто складним через необхідність написання великої кількості шаблонного коду та створення інтерфейсів для базових операцій.
    

4\. Наскільки простіше тепер змінити БД або фреймворк?
------------------------------------------------------

Завдяки впровадженню **Dependency Inversion (DIP)**, зміна зовнішніх компонентів стала максимально простою:

*   **Зміна БД:** Оскільки всі інструменти домену спілкуються з базою виключно через інтерфейси (напр., UserRepositoryInterface), для переходу з PostgreSQL на MongoDB нам взагалі не доведеться чіпати бізнес-логіку. Ми лише створюємо новий клас MongoUserRepository у шарі Infrastructure.
    
*   **Зміна Фреймворку:** Якщо ми вирішимо замінити Express на Fastify або NestJS, ми змінимо лише шар Presentation. Вся бізнес-логіка в Application та Domain залишиться стабільною.
    

5\. Вибір доменної моделі: Anemic Domain Model
----------------------------------------------

Для даного домену ми обрали **Anemic Domain Model**.

**Чому так?**: У нашому рішенні сутності (User, Booking) виступають переважно структурами для зберігання даних і не мають вбудованої поведінки. Уся логіка (валідація, хешування, збереження) винесена в оркестратори - Domain Factories та Use Cases.

**Переваги такого підходу для нашого модуля:**

*   **Процедурна простота:** Процеси є лінійними (перевірка унікальності -> хешування паролю -> збереження в БД). Розміщення цих кроків прямо в Use Case робить код дуже прозорим і легким для читання.
    
*   **Уникнення складних залежностей:** Наші процеси сильно зав'язані на взаємодію з БД. Щоб зробити модель багатою, довелося б передавати інструменти для хешування чи перевірки дублікатів безпосередньо в сутності, що порушило б принцип єдиної відповідальності та "чистоту" домену.
    
*   **Швидкість розробки:** Для CRUD-операцій та базової бізнес-логіки винесення функціоналу в сервіси (Use Cases) дозволило зберегти сутності чистими та уникнути надмірного ускладнення системи.
    

Висновок
--------

Проведений рефакторинг успішно трансформував жорстко зв'язану структуру застосунку (Fat Controller) у гнучку модульну систему, побудовану за принципами Clean Architecture. Завдяки чіткому розподілу на шари та використанню інверсії залежностей (DIP), нам вдалося повністю ізолювати бізнес-логіку від технічних деталей реалізації, таких як фреймворк Express чи база даних PostgreSQL.

Хоча впровадження такої архітектури збільшило загальний обсяг коду та ускладнило навігацію, це повністю виправдовуються здобутими перевагами. Проєкт отримав стовідсоткову тестованість бізнес-сценаріїв без підняття реальної БД, а також гарантію того, що заміна будь-якого зовнішнього інструменту не вимагатиме переписування ядра системи.

Використання Anemic Domain Model у поєднанні з Use Cases виявилося найбільш практичним рішенням для поточного функціоналу. Воно дозволило зберегти лінійність процесів і уникнути штучного перевантаження сутностей (Entities) зайвими залежностями по типу бібліотек чи підключень до бази даних.

У результаті лабораторної роботи ми отримали не просто працюючий код, а масштабовану, стійку до змін та професійно організовану архітектуру, яка повністю відповідає сучасним інженерним стандартам розробки програмного забезпечення.
