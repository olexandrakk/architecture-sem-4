## Актори
* **Адміністратор (Admin):** користувач з найбільшими правами власності. Може додавати/редагувати дані з приводу сеансів та видаляти дані будь-якого користувача.
* **Користувач (User):** авторизований користувач. Може змінювати особисті дані у своєму профілі та переглядати інформацію з приводу куплених квитків.
* **Гість (Guest):** неавторизований користувач. Може зареєструватися або залогінитися та переглядати відкриту інформацію.

## Сценарії використання (Use Cases)

### UC-1: Реєстрація та Вхід у систему
* **Актор:** Гість
* **Передумови:** Гість не має активної сесії.

**Основний сценарій:**
1. Гість відправляє `POST` запит на реєстрацію або логін із необхідними даними (email, пароль).
2. Система валідує дані (перевіряє формат, перевіряє наявність у базі або хешує новий пароль).
3. Система генерує JWT-токен із прив'язкою до ролі (User або Admin).
4. Система повертає статус `200 OK` (або `201 Created` для реєстрації) разом із токеном.

**Альтернативні сценарії:**
* **2a. Невалідні дані / Помилка входу:** Якщо при вході в сесію пароль або логін неправильний, або якщо при реєстрації email вже існує, система повертає відповідний статус помилки (`400 Bad Request`, `401 Unauthorized` або `409 Conflict`).

---

### UC-2: Перегляд доступних сеансів
* **Актор:** Гість, Користувач, Адміністратор
* **Передумови:** Жодних (відкритий ендпоінт).

**Основний сценарій:**
1. Актор відправляє `GET` запит для отримання списку сеансів.
2. Система формує список актуальних сеансів із бази даних.
3. Система повертає статус `200 OK` та список у форматі JSON.

**Альтернативні сценарії:**
* **2a. Немає сеансів:** Якщо активних сеансів немає, повертається порожній масив `[]`.

---

### UC-3: Купівля квитка на сеанс
* **Актор:** Користувач
* **Передумови:** Користувач успішно авторизований.

**Основний сценарій:**
1. Користувач відправляє `POST` запит з ID сеансу.
2. Система перевіряє валідність JWT-токена.
3. Система перевіряє наявність вільних місць на обраний сеанс.
4. Система створює запис про квиток і прив'язує його до ID користувача.
5. Система повертає `201 Created` та деталі квитка.

**Альтернативні сценарії:**
* **2a. Неавторизований доступ:** Токен відсутній або прострочений — `401 Unauthorized`.
* **3a. Місця відсутні:** Якщо вільних квитків немає, система повертає `409 Conflict`.

---

### UC-4: Перегляд історії куплених квитків
* **Актор:** Користувач
* **Передумови:** Користувач авторизований.

**Основний сценарій:**
1. Користувач відправляє `GET` запит на ендпоінт профілю для отримання квитків.
2. Система зчитує ID користувача з токена.
3. Система знаходить у базі всі квитки цього користувача.
4. Система повертає `200 OK` та список квитків.

**Альтернативні сценарії:**
* **2a. Неавторизований доступ:** Токен відсутній або прострочений — `401 Unauthorized`.

---

### UC-5: Редагування особистих даних профілю 
* **Актор:** Користувач
* **Передумови:** Користувач авторизований.

**Основний сценарій:**
1. Користувач відправляє `PUT` або `PATCH` запит з новими даними.
2. Система зчитує ID з токена і перевіряє, чи оновлює користувач саме свій профіль.
3. Система валідує нові дані та зберігає їх у базі.
4. Система повертає `200 OK` з оновленим профілем.

**Альтернативні сценарії:**
* **2a. Неавторизований доступ:** Токен відсутній або прострочений — `401 Unauthorized`.
* **3a. Пошта вже зайнята:** При спробі зміни пошти на ту, яка вже зареєстрована на іншого користувача, система повертає `409 Conflict`.

---

### UC-6: Додавання та редагування сеансів
* **Актор:** Адміністратор
* **Передумови:** Користувач авторизований і має роль `admin`.

**Основний сценарій:**
1. Адміністратор відправляє `POST` (створення) або `PUT` (редагування) запит з даними сеансу.
2. Система перевіряє токен та роль адміністратора.
3. Система зберігає або оновлює дані в базі.
4. Система повертає `201 Created` або `200 OK`.

**Альтернативні сценарії:**
* **2a. Недостатньо прав:** Звичайний користувач отримує `403 Forbidden`.

---

### UC-7: Видалення даних користувача
* **Актор:** Адміністратор
* **Передумови:** Користувач авторизований з роллю `admin`.

**Основний сценарій:**
1. Адміністратор відправляє `DELETE` запит з ID користувача.
2. Система перевіряє права адміністратора.
3. Система видаляє дані користувача з бази.
4. Система повертає `200 OK` (або `204 No Content`).

**Альтернативні сценарії:**
* **2a. Недостатньо прав:** Якщо запит намагається виконати звичайний користувач або гість, система повертає `403 Forbidden` або `401 Unauthorized`.
* **3a. Користувача не знайдено:** Якщо передано неіснуючий ID, повертається `404 Not Found`.
